# Welcome to MKG: a small Design System via Storybook
(This example is based on the material available at [**Storybook/Tutorials**](https://storybook.js.org/tutorials/).)

#### Design System: yet another modern buzzword?

At first glance, one is reminded of the well-known concept of design patterns, which offer an abstract approach for solving problems on a more technical or software architecture-specific level.

In the meantime, with design systems a well thought-out organization of existing and recurring UI components, color compositions, icons, and even layout standards has been established. The term design system stands for a professional pattern to systematically organize and simplify (partial) reuse of already existing UI front-end systems. In other words: to save working time!

Abstractly, design systems could be seen as a pattern for the following problem:

Given a set of UI components and guides derived from several projects. Is there a rough structure that I can follow to bundle all my work as a standalone system with the focus on reuse?

cf. with the motivation provided by


 [**Storybook/Tutorials**](https://storybook.js.org/tutorials/design-systems-for-developers/react/en/introduction/)
 
 [**Understanding and Supporting the Design Systems
Practice**](https://arxiv.org/pdf/2205.10713.pdf)

NOTE: One big part of a usable Design System is a documentation, which is consitent with the current state of the underlying projects. And we all know: nobody likes to write documentations (exceptions confirm the rule). 
Therefore, developers and UI-designers need a tool, which increases the motivation to work regulary on the style-guides.

Inspired by this aspect, we consider [**Storybook**](https://storybook.js.org/):

Official description copied form Storybook:
"Storybook helps you build UI components in isolation from your app's business logic, data, and context.
That makes it easy to develop hard-to-reach states. Save these UI states as **stories** to revisit during development, testing, or QA.

Browse example stories now by navigating to them in the sidebar.
View their code in the `stories` directory to learn how they work.
We recommend building UIs with a [**component-driven**](https://componentdriven.org) process starting with atomic components and ending with pages."
